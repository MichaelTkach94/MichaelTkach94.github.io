Скиллы для агентных систем

ПРО СКИЛЛЫ ДЛЯ АГЕНТНЫХ СИСТЕМ

Когда долго работаешь с агентом над проектом, довольно быстро появляется одна и та же проблема:
ты либо снова и снова объясняешь одно и то же, либо повторно ищешь в проекте уже найденные вещи, либо заново воспроизводишь процесс, который уже один раз нормально настроил.

По сути, у этого есть несколько уровней решения.

1. Сохранить наработки в файл
Самый базовый вариант: просто складывать полезные находки, инструкции и процессы в отдельный файл, чтобы потом к нему вернуться.

2. Сохранить сам чат
Тоже рабочая история: если в каком-то чате ты уже дошёл с агентом до хорошего результата, можно потом вернуться именно в этот диалог и продолжить оттуда.

3. Вынести постоянные правила в AGENTS.md / CLAUDE.md
Это уже уровень общих инструкций для агента.
Такой файл подгружается в начале работы и задаёт базовые договорённости: как устроен проект, какие есть правила, что считать нормой, какие команды запускать и так далее. Это хорошо подходит для вещей, которые нужны почти всегда.

4. Использовать skills
И вот это, на мой взгляд, самый интересный уровень.

Если у тебя есть не постоянное правило, а повторяемый сценарий, его уже лучше выносить не в общий инструкционный файл, а в отдельный skill.

Например, ревью кода.
Очевидно, ревью нужно не в каждой задаче. Если описание процесса ревью засунуть в AGENTS.md / CLAUDE.md, то это будет просто лишний шум в контексте: информация будет загружаться всегда, даже когда она вообще не нужна. В какой-то момент это может даже мешать: агент начнёт слишком рано уходить в ревью, хотя ты этого не просил.

Skill решает это аккуратнее.
Его можно воспринимать как отдельный навык агента: у него есть описание, по которому агент понимает, когда этот навык нужен, а дальше уже раскрывает его содержимое и использует нужные инструкции, шаги, ресурсы и скрипты только в подходящий момент. У Claude Code и Codex skills именно так и устроены: это расширения для повторяемых workflows, которые можно вызывать явно или доверять агенту выбрать их самому по ситуации.

За счёт этого получается очень удобная схема:

— общий контекст остаётся чистым;
— постоянные правила живут в AGENTS.md / CLAUDE.md;
— редкие, но важные сценарии уезжают в skills;
— агент получает много дополнительных возможностей, но не тащит всё это в голову одновременно.

У меня сейчас на проекте, например, есть skills на такие задачи:

— сборка Postman-коллекций;
— code review;
— написание функциональных тестов;
— разворачивание бэкапов;
— сбор новых знаний по коду и сохранение их в базу знаний проекта.

То есть skill — это, по сути, способ один раз упаковать удачный процесс работы с агентом, чтобы потом не собирать его заново в каждом новом чате.

Для меня это стало следующим шагом после файлов, сохранённых чатов и общего AGENTS.md / CLAUDE.md:
не просто хранить знания, а оформлять повторяемые рабочие процессы в переиспользуемые навыки.
